# Phase 1: Tier 1 MVP (Quick Verdict) - Context

**Gathered:** 2026-05-26 (auto mode)
**Status:** Ready for planning

<domain>
## Phase Boundary

Ship a standalone, installable Claude Code plugin that, on `/reporecon <idea>` or matching trigger phrases, returns a green/yellow/red verdict block in ≤90 seconds based on `gh api` metadata only. No clones, no WebSearch, no source inspection — those are Phase 2. The plugin's trust-critical machinery (404 verification gate, 5-axis rubric with mechanically derived verdicts, idea sharpening, confirmation-bias guard, mechanical staleness) ships in this phase because they are the product's promise, not enrichment.

Out of scope for this phase: WebSearch discovery, shallow clones, vapor heuristic, evidence file-path citations, negative-space section, marketplace submission, README polish, examples directory.
</domain>

<decisions>
## Implementation Decisions

### Plugin Packaging
- **D-01:** Single-plugin self-marketplace layout — repo root contains `.claude-plugin/plugin.json` + `.claude-plugin/marketplace.json`, with the marketplace source set to `./` and `strict: true`. The repo IS the marketplace for v1.
- **D-02:** `package.json` at repo root supplies marketplace search metadata (name `reporecon`, version `0.1.0`, description, author placeholder, MIT license, repository URL).
- **D-03:** Skill at `skills/reporecon/SKILL.md` with `references/` and `scripts/` subdirectories. Manifest schemas referenced via `https://json.schemastore.org/claude-code-{plugin,marketplace}.json` for IDE validation.
- **D-04:** Plugin name reserved early via Phase 1 commit; trigger collision check against built-ins is a Phase 3 gate, not Phase 1.

### Skill Authoring
- **D-05:** `SKILL.md` ≤ 150 lines. Progressive disclosure: orchestration logic inline, detailed rubric / query patterns / report template loaded on-demand from `references/`.
- **D-06:** Triggers: explicit `/reporecon <idea>` slash command + natural phrases (`"is there already a tool that does X"`, `"validate my idea"`, `"does this exist on github"`). Listed in SKILL.md frontmatter.
- **D-07:** Skill orchestrates: preflight → sharpen → generate queries → discover (gh search) → verify → judge-lite → report. Each step calls helper scripts (deterministic) or LLM (judgment) — never both for the same concern.

### Discovery & Verification
- **D-08:** Tier 1 uses **only** `gh api search/repositories` — no WebSearch. Avoids the 10/hr search-code limit and keeps Tier 1 fully deterministic. Per-query rate budget tracked against the 30/min search bucket.
- **D-09:** 5 queries generated by the LLM from the sharpened idea: (1) literal, (2) synonym-shifted, (3) outcome-framed, (4) tech-stack-framed, (5) adjacent-domain. Generated in a single call to keep latency low.
- **D-10:** Dedup across the 5 query result sets by `full_name`. Top 5 unique candidates by combined rank (rank-sum across queries) feed verification.
- **D-11:** Every candidate is verified via `gh api /repos/{owner}/{name}` and tagged with metadata: `stargazers_count`, `pushed_at`, `archived`, `default_branch`, `language`. Any 404 drops the candidate entirely. No URL appears in any report output without a 200 OK timestamped within this run.
- **D-12:** Preflight: `gh auth status` + `gh api rate_limit`. If unauthed or remaining < 50, abort with a one-line actionable error pointing the user at `gh auth login`. Print the rate budget the run will consume.

### Idea Sharpening
- **D-13:** Sharpening restates the user's idea as: `<one-sentence what/for-whom/how> + <3-5 differentiator keywords>`. Single LLM call, temperature pinned to 0 (or lowest available).
- **D-14:** Proper-noun guard: domain-specific terms (anything matching `[A-Z]{2,}` or capitalized multi-word phrases) are extracted verbatim into a "preserved terms" list and re-inserted into the sharpened sentence and the query templates. Prevents the model from paraphrasing NDIS → "disability scheme."
- **D-15:** Sharpened statement is the first thing shown in the report header so the user can spot misinterpretation immediately.

### Judgment Rubric (Tier 1 — metadata-only mode)
- **D-16:** 5-axis integer rubric: `core_function`, `target_audience`, `scope`, `approach`, `activity`. Each scored 0-3 per candidate.
- **D-17:** Judge prompt instructs: temperature 0, one candidate per call (no batching — batched calls drift), explicit "the user wants their idea to be novel; resist this," output as JSON `{axis_scores, rationale, candidate_verdict}`.
- **D-18:** Tier 1 verdict per candidate is `LIKELY_MATCH` / `WORTH_INSPECTING` / `UNRELATED`, derived mechanically: thresholds documented in `references/judge-rubric.md`. Full 5-level verdict (EXACT/SIGNIFICANT/PARTIAL/SUPERFICIAL/VAPOR) is **Phase 2** — Tier 1 must not claim ≥ PARTIAL because Tier 1 has no clone evidence.
- **D-19:** Overall run verdict 🟢 / 🟡 / 🔴 is derived from the highest per-candidate verdict: `LIKELY_MATCH` → 🔴; `WORTH_INSPECTING` only → 🟡; all `UNRELATED` → 🟢.
- **D-20:** Devil's-advocate re-judge: only on 🟢 verdicts where any axis score ≥ 2. Re-judge runs a second pass with framing reversed; mismatch downgrades 🟢 → 🟡. Budget-tuned: at most 2 re-judges per run.

### Mechanical Heuristics (Tier 1 scope)
- **D-21:** Staleness implemented in bash in `scripts/staleness.sh`. Emits badge tags: `archived`, `stale-12mo` (pushed_at < now-12mo), `solo-stale-6mo` (single-contributor + pushed_at < now-6mo). Vapor heuristic is **Phase 2** because it requires clone inspection.
- **D-22:** Badges surfaced in report next to candidate URL; never auto-downgrade the verdict (the fork-it case is valid signal, not noise).

### Reporting
- **D-23:** Report written to `./reporecon-reports/YYYY-MM-DD-<slug>.md`. Directory created if absent. Slug derived from sharpened sentence (kebab-case, ≤40 chars).
- **D-24:** Report header: sharpened statement, preserved terms list, overall verdict badge, timestamp, gh rate budget consumed.
- **D-25:** Per-candidate block: `full_name` URL, "verified at {ISO timestamp}", per-axis scores, Tier 1 verdict, staleness badges, one-line rationale from judge JSON.
- **D-26:** Footer: "Want deep inspection? Run with `--tier2` (clones top WORTH_INSPECTING candidates, judges with file-path evidence)." Tier 2 trigger is documented but disabled in Phase 1.

### Helper Scripts (Tier 1 scope)
- **D-27:** `scripts/preflight.sh` — `gh auth status` + `gh api rate_limit`, prints budget, exits non-zero on failure.
- **D-28:** `scripts/gh-search.sh <query>` — wraps `gh api -X GET search/repositories -F q="<query>" -F per_page=10`, jq-normalizes to `{full_name, description, stars, pushed_at, archived, language, url}`.
- **D-29:** `scripts/verify-repo.sh <owner/repo>` — `gh api /repos/...`; returns normalized metadata JSON or exits non-zero on 404.
- **D-30:** `scripts/staleness.sh <metadata-json>` — emits space-separated badge tags or empty string.
- **D-31:** All scripts pure POSIX bash 4+, `set -euo pipefail`, jq for parsing. macOS toolchain note documented in SKILL.md preflight section but enforcement deferred to Phase 3 README.

### Determinism & Testing
- **D-32:** Temperature pinned to 0 in every LLM call where the host supports it; otherwise the prompt explicitly says "respond deterministically."
- **D-33:** Golden tests in `tests/golden/` — 3 fixtures: saturated domain (`todo-cli`), genuinely-novel domain (`obscure-niche`), ambiguous domain (`llm-eval-dashboard`). Each fixture is `{idea: "...", expected_verdict_band: "🟢|🟡|🔴"}`.
- **D-34:** Tier 1 must produce the same verdict band across 3 consecutive runs on each golden fixture. Stability is the Tier 1 ship gate, not exact axis-score match.
- **D-35:** Smoke runner: `tests/run-goldens.sh` invokes the skill non-interactively against each fixture, asserts band match × 3 runs. Wired into a single GitHub Actions workflow as part of Phase 1 (CI scaffold) — submission gating is Phase 3.

### Out-of-Scope-for-Phase-1
- **D-36:** Defer to Phase 2: WebSearch expansion, shallow clones, `safe-clone.sh`, vapor heuristic, file-path evidence citations, EXACT/SIGNIFICANT/PARTIAL/SUPERFICIAL/VAPOR verdicts, negative-space "your angle" section, prompt-injection delimiters around cloned content, planted-fixture goldens.
- **D-37:** Defer to Phase 3: README polish, demo gif/asciinema, dogfooded examples, LICENSE polish (Phase 1 commits a minimal MIT LICENSE; Phase 3 finalizes), marketplace submission.

### Claude's Discretion
- Exact CSS-equivalent prose tone in `references/report-template.md` (clear, terse, no emoji beyond the one verdict badge).
- Filename slug edge cases (collision suffix scheme).
- Whether to fail open or fail closed on `gh api rate_limit` returning malformed JSON (recommend: fail closed with actionable error).
- Re-judge prompt phrasing for D-20.
